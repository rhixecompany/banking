# enhance-pages.plan.md

## Purpose

This plan file documents the implementation plan to enhance all Next.js pages under `./app` in this repository. It is the canonical plan used to coordinate small, reversible changes that:

- Audit and update pages grouped by route layout (order: `(auth)`, `(admin)`, `(root)`, then `app/page.tsx`).
- Consolidate and validate reusable dynamic generic layout components under `./components/layouts`.
- Harden Server Actions, DAL, tests, and client stores; enforce repo conventions (Server Actions: `use server`, early `auth()`, Zod schemas, `{ ok: boolean }` return shape).
- Refactor orchestration scripts in `./scripts` into TypeScript (ts-morph) with `--dry-run` support and thin shell wrappers.
- Implement `scripts/ts/mcp/mcp-runner.ts` and supporting helpers to discover, catalog, and manage MCP servers; update `.opencode/mcp_servers.json` and `.opencode/opencode.json` as needed.

This file is the first committed artifact for the work and contains the discovery snapshot, batch plan, and verification checklist.

## Assumptions

- You approved creating this plan file and starting implementation (2026-04-20).
- Secrets and local test env are available in `.env.local` (DATABASE_URL, ENCRYPTION_KEY, NEXTAUTH_SECRET).
- Branching strategy: one PR per route group (auth, admin, root, home).
- Commits that update manifests (.opencode/mcp_servers.json, .opencode/opencode.json) are allowed and will be committed to branch-specific PRs.

## First-phase deliverables (order of execution)

1. Create per-repo discovery artifacts (dry-run):
   - `.opencode/reports/pages-map.json` — authoritative page map (generated by script)
   - Commit `enhance-pages.plan.md` (this file) to branch `feature/enhance-pages/plan-only`.

2. Implement Auth group improvements (route group `(auth)`) in small batches (≤7 files per batch). Example first batches are included below.

3. Implement Admin group, then Root group, then Home page (app/page.tsx) using the same batch pattern.

4. Concurrently refactor scripts to TypeScript and add `scripts/ts/mcp/mcp-runner.ts` with dry-run support. Only apply changes after dry-run validation.

5. After all pages are updated, run full validations: `npm run format && npm run type-check && npm run lint:strict && npm run verify:rules && npm run test:browser && npm run test:ui` (E2E requires seeded DB).

## MCP runner & manifest workflow (dry-run first)

1. Discovery (dry-run): `docker mcp gateway run --profile adminbot` (wrapped by `scripts/ts/mcp/mcp-runner.ts --dry-run`) to detect running MCP servers.
2. Reconcile results with `.opencode/mcp_servers.json`. Add missing servers to the manifest (dry-run), and backup the original manifest to `.opencode/backups/`.
3. Remove listed servers (per policy) only after dry-run validation. Targeted removals (example): `mcp/context7`, `mcp/fetch`, `mcp/filesystem`, `mcp/memory`, `mcp/next-devtools-mcp`, `mcp/playwright`, `mcp/sequentialthinking`, `mcp/time`, `mcp/youtube-transcript` (match by name + sha when available).
4. Install remote NPX-based MCP tools as requested and update `.opencode/opencode.json` accordingly.

All MCP operations must support `--dry-run` and create timestamped backups before mutating manifests.

## Verification checklist (per-batch, pre-PR)

- Confirm batch edits touch ≤7 files (if >7, create a plan file per AGENTS.md).
- Run ts-morph scripts in `--dry-run` to preview code mods.
- Update Zod schemas under `lib/schemas` for any server actions added/changed.
- Grep for `"use server"` and confirm server actions include it where required.
- Add/Update unit test stubs for changed components (do not run them until full-site done).
